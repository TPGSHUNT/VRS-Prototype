2. The five things future-you needs to internalize

A. The demo audience is IT, not business

Ken's May 4 "two-cent" email is the single most actionable insight in the entire correspondence: "The people you will demo'ing to are IT people. To them VRS is a burden. They don't care what it does or what benefit it brings to the company, they care about being able to support it. What IT wants is to get rid of Oracle Forms and Reports … If you can convince them that you can convert the existing forms and reports in a reasonable amount of time and for a reasonable amount of money, then I think they will be interested — that is their pain point."

This re-centers the demo. Business value (Vera, bubble field, opportunity surfacing) is still in the pitch — but the load-bearing claim is "we can convert your Oracle Forms inventory in reasonable time and money." Khari (DG's main technical contact, per the April 30 Khari-loop email) explicitly expects to see a fully working AI-converted form at the demo — not a mock. Plan accordingly.

B. VRS isn't being modernized — it's being preserved

Per Ken's May 8 message: "Everything is being done so that VRS itself does not have to change. It will continue to do things the same way for the foreseeable future." The active DG project is Lawson AP → Oracle ERP. The Vendor Master bifurcation is being finessed under the existing materialized view specifically so VRS doesn't notice. RSL stays in Lawson. The rebuild we're proposing is TPG-driven, not DG-driven. This needs verification with Khari before the architecture doc's tone gets locked.

The VRS schema lives inside the RSL Oracle instance. The sister schemas VRS reads (Sales, Receipts, Drop Ship, master MVs) are intra-instance, not external. The rewrite is "schema extraction from a shared Oracle instance," not "lift a standalone app onto Postgres." Every cross-schema read becomes a federated read in the new architecture. This is fundamental and is already documented in 09-current-architecture.html.

C. FPA-encoded vs FPA-operating

The system encodes VRS_FPA_SUPERVISOR as the most-privileged MDSE role (36 M forms, exclusively owns Cycle Close, POG, Forecast ID Management, MDSE Rebate Type setup, DG Racing, Consolidate Vendors). But Ken explicitly says FPA abandoned the system years ago and de facto support has fallen to him.

For the rebuild this is a design fork: honor the encoded model and assume re-staffing, or redistribute FPA's authorities to who's actually doing the work. This question is now with Ken (see §4) but a guess at the answer is needed before MDSE-side roles can be modeled in the prototype.

D. Ground truth for security is now on disk

docs/ken/DG_Roles_flat.csv has 15 security groups × 90 forms with M/R/N values. This obsoletes any role inference based on prose. Key facts: AP-side has 7 groups (APP_USER, MANAGER, ADMIN, ADMIN_RO, DAMAGES, SUPERVISOR, MDSE_VENDOR_CONTACTS), MDSE-side has 8 (FPA_SUPERVISOR, FPA_USER, MDSE_BUYER, MDSE_USER, MDSE_DMM, MDSE_SVP, MKTING, PRICING). DMM and SVP are identical at form level — the Buyer→DMM→GMM/SVP hierarchy is enforced by Move Forward business logic, not security.

Period-end Review/Approve/Finalize is gated to VRS_MANAGER + VRS_ADMIN only — analysts cannot do period close, but they can do Batch-on-Demand. This is now baked into the role matrix.

E. Glossary discipline matters

Three terms get confused: Rebate Program (AP side, generates earnings), Agreement (MDSE side, generates forecast, can be Portal-submitted), and Contract (multiple meanings — legal doc, subsystem-specific data carrier, MDSE grouping mechanism for portal-bundled agreements). MDSE Agreements are not a complete picture because subsystem-driven rebates (NSA, S5S5, DG Media, BOPIS, etc.) have no MDSE counterpart. Any UI label or Vera response that conflates these will lose Ken's confidence in five seconds.

3. Pending code changes — recommended priority

User has explicitly not authorized any of these yet. This is a punch list, not a plan. Discuss before acting.

P1 Demo-visible — do before the next pitch

  • Role picker (current model is click-a-role, not login — user confirmed this is intentional and better for demo flow):
    • Add a Delegate option, or a "logged-in-as Buyer X, delegated to Buyer Y" indicator. Delegates are proxies for Buyers; FPA is delegated for all Buyers in practice.
    • Add FPA Supervisor as a selectable role (owns Cycle Close + MDSE config). Pending the FPA-encoded-vs-operating question with Ken — if Ken says redistribute, fold differently.
    • Add a "this would be SSO in production" note next to the role picker so it doesn't read as naive to Khari (who Ken says is security-sensitive).
  • Shell behavior corrections per the new matrix:
    • Audit log: visible to every role, not just APM (Ken-fix)
    • AP-stage agreement approval: routes to APA, not APM (Ken-fix)
    • Period-end Review/Approve/Finalize: visible only on APM (and Admin) shells; APA shell shows it as N
    • Batch-on-Demand: available on APA shell (Ken-fix)
    • Reports/exports: split by AP vs MDSE side; some shared
    • Renewal alerts: today's behavior is Buyer Dashboard only. Surface them on the APA shell with an "enhancement proposal" tag (Ken thinks AP should see them too)
  • Bubble field axes (Ken's May 4-5 guidance):
    • Remove or de-emphasize estimatedValue as a general axis — it only exists in the S5S5 subsystem and is misleading as a default
    • Add Merch Type × Earnings as an axis preset — Ken: "They do think a lot in terms of Merch Types"
    • Vendor aggregation: group by Vendor Name, not Vendor # (one "vendor" often has multiple AP #s)
    • Keep horizontal = contract size; vertical = Forecast vs Actual but annotate that MDSE Forecast is structurally incomplete (NSA not included)

P2 Data fidelity — do before Ken kicks the tires

  • Seed data:
    • Vendor identities: carry both AP # (VARCHAR(9), digits-only, from Lawson) and IP # (NUMBER(5), from Island Pacific). AP # is canonical on AP side; IP # on MDSE side.
    • Earnings: source data is negative ("vendor owes DG"). Normalize on ingest (recommended; render the sign in UI) or honor the legacy sign convention. Current schema sums finalEarnings and assumes positive — this is wrong against real data.
    • Two date pairs per Rebate: Rebate Begin/End vs Extract Begin/End. If schema only has one, Margin calculations break (Extract usually starts earlier than Rebate so PMU and Margin can both compute in the first active period).
    • REBATE_ID is NUMBER(10) sequence — VRS is system of record. Confirm seed uses sequence-shaped IDs, not UUIDs.
  • Glossary / labels in UI: enforce Rebate Program vs Agreement vs Contract distinction wherever they surface. The shell, the bubble tooltips, Vera prompts, anywhere.

P3 Polish — do when time allows

  • Tobacco subsystem: gate writes to VRS_SUPERVISOR+. Whether APA gets R or N is pending Ken (see §4).
  • Vendor Portal users: tag them as "Vendors, not DG associates" anywhere they appear in the UI. DG controls portal logins, but the users themselves aren't DG employees.
  • Source codes (B, C, D, E, F, N, P, Q, R, S, T, X): if the prototype shows any Rebate Type / Source dropdowns, the real list is in Ken_answers_1.txt.
  • Frequencies: Period, Quarter, Weekly, End-of-Rebate, plus user-defined (treated as End-of-Rebate). Most vendors are set up as Deduct.
  • Categories: ~170 in production (Ken's list in Ken_answers_1.txt). The over-large count is a known legacy artifact — MDSE creates a category every time an Ad Coop is set up. Worth a tooltip if Categories are surfaced.
