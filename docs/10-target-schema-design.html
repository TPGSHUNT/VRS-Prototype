Dollar General · Vendor Rebate System

Target Schema Design — v0.1 (draft)

2026-05-06 · TPG Partners · Working draft. Decisions marked REC are recommendations awaiting Ken's review.

This document is a draft of the target Postgres schema for the new VRS. It is informed by Ken's overview docs (AP Side, AP Calc, Forecast Engine), the FDD V4.1 (2010 foundational), the Toad ERD exports, and the live data CSV samples. Where I'm certain, decisions are marked IN / OUT / COLLAPSE. Where I'm not, they're marked PENDING with the question that needs answering.

What this is not
Not Prisma code yet. Not a contract. The artifact is a design document for review — once we converge on the scope and shape, the Prisma schema falls out of it in an afternoon. Reviewing this catches structural mistakes a thousand times cheaper than reviewing the code does.

1. Design approach

What we keep from Ken's logical model

  • The three-level hierarchy: RebateProgram → RebateVendor → RebateVendorDept → CalculateResult
  • One calculation per (program, vendor, dept, period)
  • Source-driven AP architecture (the 11+1 source codes drive calculation method)
  • Four earnings components: PMU, Margin, Adv Coop, Other Coop — with EADJ adjustments per component
  • Rebate Type as Source × Merch Type intermediary
  • Calculation state machine (Calculated → Reviewed → Approved → Batched → Distributed)
  • Agreement state machine with multi-tier approval routing (Pre-Negotiation through Attached AP Program)
  • The MDSE Agreement ↔ AP Rebate Program bridge via the Approval Queue
  • RSL / AP / GL as the three external batch destinations

What we change — Oracle Forms denormalization, dropped

  • COLLAPSE SBT / NSBT triplication. ~60% of CALCULATE_RESULT and REBATE_VENDOR_DEPT columns are SBT_/NSBT_ duplicates of base columns with the invariant SBT_x + NSBT_x = x. Ken's own doc flags this as "a multitude of issues" and a "future maintenance nightmare." Collapse to single columns + an SBT classification dimension.
  • DROP Vestigial AADJ accrual adjustments. Per Ken: only ONE has ever been created in the entire life of VRS. The columns exist but always zero.
  • COLLAPSE Time-horizon rollup columns (cur_, ytd_, ely_, ltd_). Aggregate views over period-grain rows; compute on read.
  • COLLAPSE Review-screen rollups (PM_* = PMU+Margin, AO_* = AdvCoop+OtherCoop). Derive in views.
  • DROP class_num column. Always -1 in production data; class-level was never implemented. If class-level ever happens, restructure then.
  • COLLAPSE REBATE_VENDOR_DEPT as a calc cache. The legacy table mirrors the most-recent CalculateResult to power the Vend/Dept tab. We don't need a second persisted copy — query the source of truth and cache at the application layer if needed.

What's added for Vera + the new UX

  • analytics_summary — materialized 1010 summaries (vendor × dept × source × period)
  • users — with role enum, replaces VRS_SECURITY
  • vendor_health_signal (or similar) — cached signal computation for the bubble field's color encoding
  • vera_session, vera_message, vera_proactive_digest — conversation memory and proactive surfacing

  • Drop ~20 PM_/AO_ rollup columns (derive in views)
  • Drop ~10 cur_/ytd_/ely_/ltd_ time-horizon rollups (compute via window functions)

6. Sign convention & SBT collapse

Sign convention

Recommendation
Store earnings as positive scalars. The legacy "natural negative" (negative = vendor owes DG) is sign-flipped at ETL ingest time, with a marker that says "this is what the vendor owes." UI renders semantically (color coding for owed-vs-paid). Aggregation, sorting, and Vera reasoning are all simpler with positive numbers. Bubble field "size = earnings" needs positives anyway.

Trade-off: ETL has to flip signs on input, and downstream RSL/AP/GL exports have to flip back to legacy convention if those consumers expect negative. Cost is one ETL step; benefit is every query, view, and Vera prompt is more natural. Sign-on Ken: any downstream consumer that strictly requires negative values?

SBT / NSBT collapse

Recommendation
Drop the triplication. Add a single sbt_classification field per relevant row (likely on rebate_vendor or calculate_result — depending on whether the classification is a vendor attribute or a per-line attribute, which we don't yet know). Provide a vw_calc_result_sbt_split view that computes the SBT/NSBT split on demand for downstream consumers that need it.

Risk: if the SBT/NSBT split is consumed downstream by RSL or AP exports in a way that requires the original split values stored as columns, the view-based approach may not be sufficient. Question for Ken: do RSL/AP/GL exports require the literal SBT_/NSBT_ columns, or can they be derived?
