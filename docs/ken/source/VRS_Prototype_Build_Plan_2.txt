Governing Principle: No Throwaway Code
The prototype serves two purposes simultaneously: it demonstrates the full vision to Dollar General in a form they can interact with, and it produces code that is structurally reusable in the production build. These are not competing goals if the prototype is built on the production stack from day one.
The temptation in prototype work is to reach for shortcuts — mock data in flat JSON files, hardcoded layouts, no real authentication, no database. That approach produces a convincing demo and six weeks of throwaway work. The approach described here produces a convincing demo and six weeks of production-grade foundation.
Database Strategy
2.1  What ‘Keep the Data Model’ Means
The recommendation to keep the VRS data model means preserving the logical design as a construct — the entity relationships, the three-level hierarchy, the one-calculation-per-period rule, the source-driven architecture. It does not mean keeping the physical Oracle database.
The current system runs on Oracle Database, driven by Oracle Forms and PL/SQL. The replacement stack is PostgreSQL. These are different engines and there is no attempt to keep Oracle in place. The migration path is a schema translation plus a data migration at cutover — not an ongoing hybrid.
2.2  Prototype Database: PostgreSQL from Day One
The prototype uses a real PostgreSQL database from the first day of development. Not JSON files. Not an in-memory store. A properly structured relational database with the production schema, populated with realistic mock data.
This decision alone is what makes the prototype non-throwaway. When the time comes to populate it with real DG data migrated from Oracle, the schema is already proven, the Prisma models are already written, and the API layer is already tested against the shape of the data. The only change is the data source.

2.4  The 1010 Database
The 1010 Oracle database stays where it is. It is not migrated. What the new system builds is a read path: a nightly job that executes summarization queries against 1010 and writes the results into materialized summary tables in the VRS PostgreSQL database. The VRS application never queries 1010 directly.
This approach is correct regardless of whether the 1010 database is eventually migrated in a later phase. The materialized summary tables in VRS are a stable, fast, independently-deployable analytical layer that does not depend on 1010’s engine, schema, or availability.

What the Prototype Demonstrates
The prototype covers six demonstration surfaces. Each surface is built with real component architecture, real data fetching patterns, and real state management — the only thing that is not real is the data itself.
4.1  Role-Simulated Login
A login screen with a role selector (AP Analyst, Buyer, DMM, AP Manager, Read-Only). Selecting a role logs in as a pre-seeded mock user with that role’s permissions. After login, every surface in the prototype reconfigures itself based on the selected role. This is the single most important demonstration for DG stakeholders — it makes the role-lens concept immediately tangible.
4.2  Vendor Bubble Field
The landing page after login. A full D3.js force-simulation bubble field with mock vendor data covering realistic earnings distributions (a small number of large vendors, a long tail of small ones). All three visual encodings implemented: size, color, and pulsing ring. Filter controls for merch type, period, status, and a My Programs toggle. Clicking a bubble navigates to the vendor record.
The mock data is seeded into the PostgreSQL vendors and rebate_programs tables. The bubble field queries the database via a Next.js API route — not a JSON file. Replacing mock data with real data requires no code change.
4.3  Vendor Record — Role-Lens View
Clicking a bubble opens the unified vendor record. The record renders differently based on role: AP Analyst sees calculation status prominently; Buyer sees agreement pipeline; DMM sees approval queue items. All views are tabs on the same component — the role determines which tab is active by default and which actions are available in the context menu.
The 1010 Intelligence tab is implemented with mock analytics_summaries data showing pace-to-target as a progress bar, YoY variance as a sparkline, and tier threshold proximity as a callout. This tab demonstrates the highest-value differentiator of the platform.
4.4  Agreement Workflow (MDSE Lens)
A buyer creates an agreement through the guided wizard. The agreement is written to the agreements table with Pre-Negotiation status. Move Forward advances it through the status chain. When it reaches Pending AP Approval, the vendor’s bubble on the landing page gains a pulsing ring (demonstrating real-time state reflection). The Approval Queue surface shows the pending item and allows inline approval, which creates a rebate_programs record atomically.
4.5  Period Close Workflow (AP Lens)
The guided period-close checklist surface. Mock calculate_results data pre-seeded for several programs in various states (open, E&C run, reviewed, approved). The checklist shows real completion percentages. The Approve step allows bulk approval. The Batch step creates a batch record. Finalize closes the period. Each step updates the database and the checklist re-renders — demonstrating the workflow without requiring a real E&C engine.
4.6  Ask Vera
A persistent chat interface accessible from every screen. Vera is wired to the Claude API with a system prompt that gives her full context of the mock vendor portfolio. She can answer questions about specific vendors, calculations, agreements, and earnings pace. She can generate proactive suggestions based on the mock data. She knows which role is logged in and calibrates her answers accordingly.
In the prototype, Vera queries mock data directly via the Claude API without the full stored-procedure routing layer. In production, the routing layer is added but the chat UI, session management, and role injection are identical.

The non-throwaway test
Before writing any code, ask: would this module, component, hook, or schema survive intact into the production codebase with real data substituted for mock data? If the answer is no, redesign it. The prototype is not a proof of concept — it is Sprint 1 of the real product.
Oracle construct | PostgreSQL equivalent
NUMBER sequences for PKs | SERIAL or UUID primary keys (UUID preferred for multi-tenant)
VARCHAR2(n) | VARCHAR(n) — identical semantics
DATE with time component | TIMESTAMPTZ — explicit timezone handling
PL/SQL stored procedures | TypeScript service layer + optional PostgreSQL functions for complex calculations
PL/SQL validation logic | Zod validation in API layer + PostgreSQL CHECK constraints
Oracle sequences | PostgreSQL sequences (via SERIAL or explicit CREATE SEQUENCE)
SYSDATE / NVL / DECODE | NOW() / COALESCE / CASE — standard SQL equivalents
Oracle partitioning | PostgreSQL table partitioning (for CALCULATE_RESULT history tables)

Layer | Technology · Rationale
Framework | Next.js 15 (App Router) · TypeScript. Server components handle data fetching, client components handle interactivity. The App Router’s file-based routing maps cleanly to the role-lens navigation model. Identical to GMCC stack.
Styling | Tailwind CSS + shadcn/ui component library. Tailwind provides utility-first CSS that travels from prototype to production unchanged. shadcn/ui provides accessible, composable base components.
Database | PostgreSQL 16. Hosted on Railway or Supabase for prototype (zero-config, instant provisioning), migrated to RDS or Azure Database for PostgreSQL in production. Schema is identical in both environments.
ORM | Prisma 5. Type-safe database client that generates TypeScript types from the schema. The Prisma schema file IS the data model documentation. Migrations are version-controlled. Identical to GMCC.
API layer | Next.js API routes (Route Handlers). tRPC optional for end-to-end type safety on complex queries. The API layer is where Oracle PL/SQL logic is re-implemented in TypeScript.
Auth | NextAuth.js v5 with credentials provider for prototype (username/role selection). Azure AD SAML integration in production. The auth interface is identical — only the provider changes.
Visualisation | D3.js for the vendor bubble field. React-based, server-side data, client-side rendering. The bubble physics and encoding logic written once and reused in production.
Ask Vera | Anthropic Claude API for prototype (fast to wire up, no infrastructure). Migrated to Azure AI Foundry / GPT-4o for production if DG requires Microsoft stack alignment. The Vera service layer is provider-agnostic by design.
Job scheduling | Node-cron for prototype E&C simulation and 1010 summary jobs. Temporal.io in production. The job logic is identical — only the scheduler wrapper changes.
Deployment | Vercel for prototype (zero-config Next.js hosting). AWS ECS Fargate or Azure App Service in production.
Version control | Git / GitHub. Conventional commits. The prototype repo becomes the production repo — no fork, no rewrite.
Non-throwaway note
The auth module uses NextAuth.js with a credentials provider backed by the users table in PostgreSQL. In production, the credentials provider is replaced with an Azure AD SAML provider. The session handling, role injection, and permission checks are unchanged.
