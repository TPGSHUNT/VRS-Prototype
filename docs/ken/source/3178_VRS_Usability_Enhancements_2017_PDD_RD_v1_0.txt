Project Design Document (PDD)
Purpose
The purpose of this project is to enhance the usability of VRS system for merchandising users. This project is an amalgamation of various requests from business users to help them use the system in a more efficient way.
Project Scope 
The Project Scope is limited to the requirements listed in the Business Requirement section of this document. This list was developed by having several meetings with Business users, to understand the scope of the project. The delivery of project scope will be divided into phases as discussed in Project Phases section.
Project Phases
The project will be deployed in multiple phases spanning till the scheduled year-end freeze. The business users and IT will determine which requirements will be implemented in each phase, considering the project schedule and cost to benefit analysis. It is understood that not all requirements stated in the Requirement section can be fulfilled in these phases but a concrete effort will be done to address most of them.
The tentative dates for Implementation of these phases are as follows
Phase 1	May 5 2017
Phase 2	July 7 2017
Phase 3	Sep 8 2017
Phase 4             Nov 3 2017.

Technical Comment (Ken Banks) – this issue is the result of a limitation built into all the “selectors” used in VRS; however, most of the selectors are used in situations where the number selected is fairly small so the limitation is not an issue.  The limitation is the result of building an SQL “IN” list in a 4000 byte (max size) variable.   For stores, since the Store number is the key to the table, the limitation is somewhere between 650 and 700 stores depending on the stores selected.  This limitation is not formally tested for so when the limit is reached, the form exhibits erratic behavior and selections can no longer be made.  This was addressed by using a temporary instead of a variable to hold the list. See the Technical design section at the end of this document.

INC0124037 - Calling of the Allocation Functions when an Adjustment is made on a Discount Amt/BOGO Rebate
Description:  On the Rebate Program form, after an Adjustment is made, the appropriate Allocation function(s) are called based on source. This is done so that the Adjustment will be reflected in the Allocation data.   When the Discount Amount/BOGO Source was introduced, the invocation of the Discount BOGO Allocation Routine in the logic following the adding of an Adjustment was not done.  Therefore, when an Adjustment is made on a Discount Amount BOGO Rebate Program, it is not reflected in the Allocation data until the next time an Extract and Calculate is run.   If another Extract and Calculate is not run prior to the Closing of the period, the Allocation data will not reflect any Adjustments and if this data is used when applying actuals to an Agreement, the Actual amounts will be off by the amount of any Adjustments made.
Null SBT causing Issue on Rebate Program Forecast Form
NULL SBT Values caused a production issue when Agreements with a NULL were used in the Approval Queue to create Rebate Programs.  They were also causing an issue on the Forecast Rebate Program form when an Agreement with one of these was encountered – the form indicated that a Change had been made by asking if the Changes needed to be saved when if fact, no change had been made.  
It was initially decided that a table change would be made to Default SBT Type to N but Lane objected to this and wanted to the Buyers to specifically choose the SBT Type on Portal Agreements. The SBT being NULL is a good indicator that it needs to be set; therefore, since the Forecast Rebate Program form was already having issues with the NULL, both problems can be solved with a change to the Forecast Rebate Program to deal with NULL SBT Type more directly. 
Delete of Vendor’s User id should not be allowed if it has been used.
A production issue occurred whereby in the reprinting of a Portal Agreement, the Address disappeared. This was caused by the Vendor’s User ID being deleted from the system because the Address was tied to the User, not the Contract. It was decided to prevent the deletion of any Vendor User id if it had ever been used on a Contract. 
When an SBT Type is NULL, which occurs on Agreements coming from the Web Portal, the SBT / Non SBT Radio buttons that appear on the Vnd/Agmt tab will not have a choice made.   The lack of a choice made will indicate that the SBT Type is NULL.  In cases (sourced Agreements), where the SBT / Non SBT Radio buttons would not normally be there, the Non SBT Radio button will be turned back on but will not be selected.  So NULL is now considered a Valid value from the Forms perspective  but will prevent the form from moving forward. 
If there is an attempt to Move Forward and the SBT Type is Null, A message will indicate this and the Agreement will not be allowed to move forward.

Cooler Counts from needs to handle changes to Agreement Dates.
The Cooler Count Form is called from the “Add Wizard” when an Agreement is first Created (Add Mode) and it can also be called from the STORE_SELECT BTN’s – WHEN-BUTTON_PRESSED Trigger (Edit Mode). The issue being addressed here is that when the form is called in Edit Mode, the periods are not and cannot be adjusted to match the Agreement Dates.   This is because this is only done when the form is called from the “Add Wizard”. Note also that the form itself does not do the Inserts into the MDSE_AGMT_STORE_CNT, this is done in the Add Wizard code. 
The way the form knows that it is called in Add Mode verses Edit mode is that it looks in the Record group 'COOLER_COMMENT_RVAL' and specifically at the number of rows.   This Record Group is loaded from COOLER_STORE_COUNTS Table/Array that is passed into the MDSE_LIB’s function CALL_COOLER_COUNTS and will be empty when the form is called from the Add Wizard.  In this case, the records Group is loaded using the From and To Dates.  This does not happen in Edit Mode – the existing data in the Record Group is used: and this is the issue.  In Edit Mode there needs to be a comparison between the Agreement Dates and the first and last row in the Record group to see if anything has changed.  If so, the Record Groups needs to be adjusted accordingly.  Also, since the Inserting into the MDSE_AGMT_STORE_CNT table occurs outside of the form, code to do this will need to be added to the WHEN-BUTTON-PRESSED trigger on the STORE_SELECT BTN. 
In summary, in Add Mode the Record Group is loaded by the Cooler Count Form using the Agreement Dates and the Table is loaded after the form returns using the data in the Record Group.  In Edit Mode, the Record Group it is loaded from the table and is used as is and this is not correct when the Agreements dates have been changed.

An assumption is a statement that is believed to be true in the absence of proof or definitive knowledge.  As an example, an assumption could be “Receipts will be made against the correct purchase order.”  An assumption can also be a business rule, a statement that defines or constrains some aspect of the business. An example of such a business rule might be that a store is supplied from one and only one distribution center. The author should list business assumptions that may be included as “rules” or constraints within the system/process being developed or modified.

A dependency is an external factor outside the control of the project which must exist or be completed before or during the project.  An example of a dependency would be another system under development that must be finished in order to provide data to the system being developed.  List any dependencies and the anticipated time when the predecessor must be complete.

The process flow is the series of events required to execute a function.  Process flows are best described using  flowcharts.  The process flow should be written using business terms, events, and employee titles.  Document each decision point and the flow that stems from each decision.  As this is a business flow rather than a system flow, consider using swim-lanes rather than flowcharts to depict the process.
Authentication refers to “how” the users log into the system.  Usually this is through the operating system username and password, but the circumstances of the system might warrant a different or additional authentication procedure.  An exception might be made for a particular application that only requires a ‘pin’ number to access a feature rather than a password.
Define what each new role may or may not have access to for each of the new features identified in the Process Flow.  For each event to which a security role has access, the access may be read only, add only, add/change only, etc.  In some instances, the access may be even lower, i.e. the role might only have access to one DC’s information, or the role may have access to everything on a screen, except one field such as pay rate.  Document these requirements.  Include which job title(s) have access to each role/access.
