Overview


Purpose 

The purpose of this document is to describe the functionality of the Merchandising Vendor Rebate System (VRS).  The VRS application will allow the Dollar General Finance Planning and Analysis (FP&A) group and the Merchandising group the ability to plan, negotiate and approve rebate programs, and forecast rebate monies.  In addition, the VRS will also allow Dollar General (1) to view forecasted rebate dollars versus actual rebate dollars at the rebate program level, and (2) analyze and make more informed and precise decisions on future planning and forecasting rebate dollars based on historical data.     
The system will automate the allocation of forecasted rebate dollars by department, class or SKU using information obtained from aggregated vendor receipts files or from other sources (such as sales data) provided by the user in combination with a method for period allocation based on a default selected for Merchandising Type.
The VRS Phase 2 for Merchandising will include the following:
Has basic functionality for:
Maintaining and tracking plan rebate dollars
Negotiating and approving agreements within a rebate program
Forecasting rebate dollars
Comparing forecast versus actual rebate dollars versus plan dollars
Reporting on forecast versus actual rebate dollars versus plan dollars
Provides ability to:
Setup and edit vendor agreements through online screens
Communicate electronically to others rebate programs waiting for their approval
Automate the allocation for rebate programs based vendor receipts and/or sales
Provide flexibility to override %, $ for allocating forecasted rebate dollars
Conduct a ‘Final Lockdown’ and take a snapshot for a fiscal period
Provide FP&A and Merchandising group an overview of their own rebate programs using a dashboard
View Audit Log
Conduct variance analysis

Alignment with Business Strategies and Objectives

Dollar General would like a comprehensive Vendor Rebate System that will forecast rebates, actualize rebates, and assist with financial reporting, and management of rebates.
Dollar General requires a comprehensive Vendor Rebate System that will do the following:
Provide online applications and tools for planning, managing, and tracking rebates in a more effective and accurate manner
Provide an integrated system that will improve communications, accuracy, and productivity among Dollar General Merchandising, FP&A, and Accounting groups. 
Streamline business practices that will allow Dollar General more time to focus on analyzing and closing additional business opportunities
With respect to the Vendor Rebate System - Merchandising, Dollar General has identified the following success factors:
To relieve buyers and merchandising teams from the tedious and error prone data entry of forecast information
To streamline communication between various groups: FP&A, merchants and A/P. 
To provide an auditable rebate data repository for all rebate information. 
To standardize entry and maintenance of forecasts at the class level or SKU level
To assist in analyzing and trouble-shooting rebate program performance


Project Approach

In order to align the Vendor Rebate System project with Dollar General’s business strategies and objectives the project is dependent on the Accounts Payable Vendor Rebate System Phase 2.   This document will only cover the merchandising part of the Vendor Rebate System project.

Security

A Security function will be used to manage access to the Vendor Rebate System.  Each user will be assigned to an access group.  New employees will be assigned to an access group via the DG IT Access Form. The groups for merchandising will include DMM, BUYER, FP&A, and FP&A_SUPERVISOR.  The Security application shown below manages the configuration for these access groups.

The system will use a combination of data access and user role assignment (including as a Buyer’s delegate) throughout the system for security. Buyers (or their delegates) will have access to change forecast and agreement information based on the class to buyer assignments. 




Within each group the menu selections will be defined to allow Maintenance access, Read Only access, or No Access.  Maintenance access will allow the user to utilize the associated application to its full potential.  Read only access will allow the user to only see the data within the associated application.  No access menu selections will be ‘grayed out” indicating they cannot be accessed by the user.

For example, all merchandising access groups (and therefore users) will have “read only” access to the AP Rebate Program option since it controls Accounts Payable information and processes.

1.1 Plan Maintenance

The Create/Revise Plan tab will be used to construct a Plan from components or change the components of an existing plan. The plan cannot be created for prior fiscal years.
FP&A will mark a version of the preliminary plan to become the final Vendor Rebate Plan for a fiscal year. FP&A will export the final plan dollars from the Rebate Plan for upload into Arthur.
FP&A may also export other plans as necessary. Access to Plan Maintenance will be limited to the FP&A Supervisors through system configurations.

Creating a Plan with Components: 
When creating a new version of a plan, the user will type in the name in the “Plan Version” field and select the fiscal year of the plan.  
Plan Sources include 1) other Rebate Plans, 2) roll ups from detailed rebate forecasts, 3) existing FP&A forecasts and 4) .csv uploads.  When selecting a .csv source, the user will choose if the upload will either replace the existing values or add to (adjust) the values. 
For each source, the user will identify the months the source’s data will be built from.  By using the “Remaining open periods” selector, the user can create the Plan with data (or for upload) in the remaining unselected fiscal periods.  
For Plan, Forecast and upload data, a version of the source will be identified.  The Plan and Version options will list existing versions in the source version grid.  The .csv upload source will require the name of the file be entered.
Pressing the “Add Component” button will load the source and its detail into the Plan Components grid.
The user has the option to remove a previously selected source by highlighting the source row in the grid and pressing the “Remove Component”.
When all sources for the Plan have been defined, the user presses the “Create” or “Revise” button.  The system will build the version using the sources.  (Depending upon the volume of data, this may occur as a batch process.)
Notes: 
When creating or revising a component from a source with multiple elements, the system will copy all of the source’s current elements (FP&A adjustments, buyer adjustments and total from detail elements) into the new component elements.
When creating or revising a component from the detail roll-up, the system will archive the detail rebate forecasts and mark as used in the version being built for future reference.
The header section of the form will be cleared when the user moves between tabs on the plan maintenance form.
The upload process will update only the fiscal periods containing numeric values in the period data columns. The system will not adjust a period’s value if the adjustment column has null or no value in the input .csv file.  If a fiscal period contains zero, the system will use zero as the value for the fiscal period.
A new plan will be created as a preliminary locked plan. 
When creating or revising a plan, the components will be processed in the sequence they were added with the exception of the upload component which will be initiated by the user through the upload process.  
The upload process will not execute while the process to build a plan is scheduled or in progress. 

Revising a Plan’s Components: 

This form will also be used to revise the components of a forecast.  
When revising a plan, the version can be selected from a list presented in the “Plan Version” box.   Versions marked “Final” cannot be revised.
All sources would be available.  
The “Remove Component” in the Plan Components grid would apply to the sources being used in the current session to revise the plan.

An audit log entry will be created

1.3 Plan Upload

Uploading Plan Data

Plan data can be created or adjusted using a .csv upload.  

The upload will adhere to the following rules:
The upload will include an existing plan name/version.
Data from a .csv upload will be the FP&A adjustment element of the plan.
The file name will agree with the file name entered in the plan component.
A department is required in each record and must exist in Dollar General Merchandise hierarchy.
The upload may be applied to an unlocked “Final” plan. 
The rebate type is required and must exist in the Merchandise Rebate Type file of the Vendor Rebate System.
Vendor, rebate category and class are optional; if these fields are on the records, they must exist in Dollar General’s master files. 
If the vendor is not present, the dollars will be added to the “No vendor” vendor.
If the class is not present, the dollars will be added to the “No class” class.
If the rebate category is not present, the dollars will be added to the “No rebate category” rebate category.
The amount must be numeric or null and can be positive or negative.  If the value is zero, zero will be used in the process.  If the field is blank or null, no value will be used in the process. 
If the upload plan component option is “replace”, the forecast numbers will be replaced. If the upload plan component option is “add-to” the forecast numbers will be adjusted.
Only periods defined as comprised of upload data on the upload components will be changed.  If data exists in other columns in the .csv file, the data will be ignored. 

If exceptions are found, the upload will not occur and exceptions will be reported. The exception report will include details on the record and the exception condition to aid in resolution. FP&A is responsible for the rebate plan.  Therefore, access to the Rebate Plan upload will be controlled through the assignment of the appropriate user group to the .csv upload.

1.5 Plan – Add Adjustments

The Review and Revise Rebate Plan form provides the ability to view and add Buyer adjustments for a specific department and fiscal period to a rebate plan in the Merchandising Rebate Forecast System.  
Buyers will enter adjustments in the Buyer Adjustment column.  FP&A will enter adjustments using the .csv upload.  FP&A may view the plan data using this form.  
If the Rebate Plan version is marked “Final”, Buyer Adjustments cannot be entered.  
This process will mirror the FP&A Add Adjustment process as defined in section 7.5 FP&A Forecast – Add Adjustments.  


Using the selection options, the user can select the version of a plan, the department and fiscal month to view in the form.  The system will allow the user to enter buyer adjustments based on the departments in his/her area of responsibility.  The user can then update the adjustment for a specific rebate type.
Buyers may not enter adjustments to a locked version of the plan nor to a “Final” version of a plan.
When the buyer enters an adjustment, the Plan dollars will recalculate.

If any part of the From/To date for a forecast program falls into a previous fiscal year or previous period of the current year period then:
The period allocation for the forecast will be calculated for the entire time frame
The portion that falls into previous fiscal year will not be included in the forecast
The portion that falls into the previous period of the current year will be added to the current period
For example, if current period 4 of  fiscal year 2010 period 4 AND a forecast program comes in for Nov 2009 through Oct 2010 then (current fiscal year/period is 2010 and period 3)
Portion for Nov thru Jan will not be forecasted
Portion for P1-Feb thru P3-April will be added to the P4 - May period allocation
This will coincide with the method AP uses to actualize earnings
If the number of weeks for a given period differs from previous year (i.e. if previous fiscal year has 52 weeks and current fiscal year has 53 weeks), then the base volume will be adjusted to account for the different number of weeks between the previous and current fiscal years
If there is an extra week then the system will calculate a weekly average and add to the base volume for the period with the extra week
If there is a week less then the system will decrease the base volume for the period for the 1 less week 
ONLY APPLIES if covering for the whole period

3.0 Vendor Submitted Rebate Agreements

Today a Dollar General vendor submits rebate agreements through spreadsheets emailed directly to the appropriate buyer. The agreement spreadsheets contain information on either Clip Strips or Side Wings or information on multiple merchandise rebate types related to a specific Ad Event; i.e. End Cap, Ad Print Coop, Scan-back Ad and TPR. For new vendors, the buyer may assist with completion of the initial agreement which the vendor will use subsequently as a template to submit additional agreements directly to the buyer or the buyer may create each agreement for the vendor to approve.

With the Vendor Rebate System, a Web Based Application will be will be used by the vendors rather than the spreadsheet / email approach.  Access to the application will be through a link on the existing DG Vendor Portal page (DGPartners.com). The application will provide standardized entry of those merchandise rebate types where vendor submission is allowed and validation of pertinent data elements at the time of entry. This eliminates the need for the buyer to enter the information into the Vendor Rebate System manually if the spreadsheet / email process remained in place.

The vendor will be able to enter agreements for the following Merch Types:
Ad Print Coop 
Scanback Ad 
End Caps
Clip Strips
Side Wings

The following business rules will apply:
Prior to a vendor using the application, the buyer will provide the vendor with their IP vendor number and AP vendor number (two unique numbers are needed for authentication)
Vendors must enter a valid UPC (both 12 and 13 digit UPCs will be accommodated) or SKU. If a UPC is entered, the associated primary SKU will be added to the rebate form. If a valid non-primary SKU is entered, the system will convert the entered SKU to the primary SKU before adding it to the form.  The entered UPC will be maintained for reference purposes
For Clip Strip and Side Wings items, a comment must be entered for each item indicating the disposition of remaining product.  The system will validate the comment contains text
Checking the Agree checkbox and pressing the Submit button will signify the vendor’s approval of the contract
The agreement status will be set to ‘Pending Buyer Approval’ and will appear on the appropriate buyer dashboard based on the class to buyer mappings
Forms containing multiple merchandise rebate types will be systemically split into multiple agreements / forecast IDs but will be linked by a systemically assigned contract ID
Contact information entered by the vendor will be compared to the VRS merchandise vendor contact information and if different, capture as a comment that will be added to the agreement. 
The vendor will print the form, sign it, scan it and email to the buyer who will attach the signed copy of the printed form as the final contract. Or the vendor will print the form and send an email signifying their approval to the buyer who will attach the printed form and the email approval as the final contract.
The form will mirror the current Excel forms used.
 A/P would like to capture some form type of information about the person entering the agreement to verify authenticity of the vendor entering the agreement and the date/time of the submission.

CGI will work with the Dollar General IT group to explore the security, legal and feasibility implications to accomplish the functionality outlined in this section.

Tech notes:  Integrating Vendor Submitted Agreements with VRS Rebate Agreements
Ad Event Vendor Submitted Agreements:
A single vendor submitted Ad Event agreement may result in multiple VRS rebate agreements.  
An Ad Coop Merchandise Rebate Agreement will be created with a NS-Period source from an Ad Event which has lump sum amounts. Items on the Ad Event Vendor Submitted Agreement with no associated unit allowance or fee will be included in the VRS rebate agreement for informational purposes.  If all items on the agreement include a unit allowance, all items will be listed on the VRS rebate agreement for informational purposes.
An Ad Coop Merchandise Rebate Agreement will be created from an Ad Event with items with Ad Coop Circular Fees entered with a NS Lump Sum source. 
A Scan Back - Ad Merchandise Rebate Agreement will be created from an Ad Event with items with an associated unit allowance with a Sales source.  
An End Cap Merchandise Rebate Agreement will be created from an Ad Event with items with Display fees entered with a NS Period source. 

Clip Strip Vendor Submitted Agreements:
A single vendor submitted Clip Strip agreement may result in multiple VRS rebate agreements.  
A Clip Strip Merchandise Rebate Type will be created with a NS-Period source from a Clip Strip Vendor Submitted Agreement which has a lump sum amount. If all items on the agreement include the rate, # of Sides and # of Stores, all items will be listed on the VRS rebate agreement for informational purposes.
A Clip Strip Merchandise Rebate Type will be created from a Clip Strip Vendor Submitted agreement with items with Rate, # of Sides and # of Stores as NS-Rate source.

Side Wings Vendor Submitted Agreements:
A single vendor submitted Side Wings agreement may result in multiple VRS rebate agreements.  
A Side Wings Merchandise Rebate Type will be created with a NS-Period source from a Side Wings Vendor Submitted Agreement which has a lump sum amount.  Items on the Side Wings Vendor Submitted Agreement form without the rate and # of Stores will be included on the VRS rebate agreement for informational purposes. If all items on the agreement include the rate and # of Stores, all items will be included on the VRS rebate agreement for informational purposes.
A Side Wings Merchandise Rebate Type will be created from a Side Wings Vendor Submitted Agreement with NS-Rate source.

5.0 Dashboards

5.1 Buyer’s Dashboard 

5.1.1 Business Rules / Overview

In all instances, when the user has invoked the agreement maintenance screen or detail forecast screen, the invoked screen will display the information for the selected agreement or detail forecast.  When the user closes the invoked screen, the user is returned to dashboard. 

On the detail agreements, AP and Detail Forecast forms, the system will provide a drop down box with applicable actions.  The actions are dependent upon the status of the agreements, detail forecasts or AP rebate programs.  The user will select the action and press Save.  When no action can be taken by the buyer, the action box will have no valid selections. 

When the user selects “Refresh”, “Return to Buyer Dashboard” or “Exit”, the system will check if changes have been made and not saved.  If this condition exists, the system will display a pop-up box and ask the user “Save Changes?  If the user selects “Yes”, the changes will be saved.  If the user selects “No”, changes made on the screen will be disregarded.  

Multi-department agreements

A multi-department agreement would be managed by a buyer designated as the system owner or “primary buyer” for the agreement.  The primary buyer would be the assumed to be the merchant entering the agreement (or, where a delegate is entering the agreement, the merchant the delegate is assisting.)  The multi-department agreement will require approval of the buyers responsible for the other departments on the agreement before moving down the approval route.

A New Multi-Department Agreement:
The agreement will be moved forward in the merchant approval process by the primary buyer.  

The agreement then will appear in 2 dashboards:
on the secondary buyers’ dashboards with “Pending Buyer Action”.  The secondary buyer can take one of two actions on their segment of the agreement:
approve – which will remove the agreement from the approving secondary buyer’s dashboard
reject – which will mark the agreement as “Multi-department Rejected by Buyer” and send it to the primary buyer’s dashboard
on the primary buyer’s dashboard with “Pending Buyer Action”.  The primary buyer can take one action if the agreement requires significant change:
set to “Pending Buyer Approval” – which removes from all secondary buyer’s dashboard and sends it to the primary buyer’s dashboard for change and resubmission through the merchant approval process.

A Multi-Department Agreement Rejected by a Secondary Buyer:
If one secondary buyer rejects a multi-department agreement, the agreement will continue to appear on the other secondary buyers’ dashboards ‘Pending Buyer Action’ and now on the primary buyer’s dashboard with “Multi-department Rejected by Buyer”.  The primary buyer will resolve the issue causing the segment of the agreement to be rejected then “move forward” the agreement.  The agreement will reappear on the secondary buyer who rejected their segment of the agreement.

Move Forward
The agreement will automatically move forward in the merchant approval process when all secondary buyers have marked their segment of the agreement as approved.  The agreement will disappear from the primary buyer’s “Pending Buyer Action” section of the dashboard and continue in the merchant approval process with its next status dependent upon the type and size of the agreement (as listed in the ‘Move Forward’ section below).


5.1.2 Statistics Overview

The system will display statistics on the Buyer’s specific agreements, forecasts and AP Rebate Programs.  The status will display if at least one agreement, forecast or AP rebate program exists with the status. 

The user’s role will determine the view on the Dashboard.  A Buyer’s view will have the Buyer selected and, using the criteria noted below, optionally the Department and Merchandise Rebate Type may also be selected.  Assistant Buyers and MA’s will be required to select the Buyer from a list of Buyers who have the user defined as the Buyer’s delegate.  

If the buyer has responsibility for multiple departments, the user will have the option to select a department or series of departments.  If the buyer has responsibility for multiple departments and the user does not select a department, all departments will be the default.  If the buyer has responsibility for a class or classes within a department or a single department, the department will be pre-filled and grayed out.  If the buyer has responsibility for classes within a department, the system will use the classes within the department as selection criteria.  (The department will be displayed in the selection criteria). 

The buyer may also select a merchandise type or multiple merchandise types.  If the buyer does not select specific merchandise type, all merchandise types will be the default. 

The buyer will highlight a status and press “View Detail” to proceed to a list of agreements, forecasts or AP Rebate Programs with the status and take action. 

The buyer will have the option to review new deals and changes made to detailed forecasts not in the most recent rollup using the “New Deals and Revisions” button.

The top row in the ‘Detailed Forecasts’ column will show the number of forecasts with the status of ‘Forecast not Submitted to FP&A’ that are within 30 days of the begin date so they can be worked as a priority item. The top row of the ‘AP Rebate Programs Column’ will show the number of with a status of ‘Pending AP Approval’ that are within 15 days of the begin date. These 2 special statuses will provide a way for the buyer to identify ones that might need to be worked before others to ensure they are ready when earning are scheduled to begin. The 30 days and 15 days are configurable options. 

Note: Assistant Buyers’ and MAs’ Buyer delegation will be done through the Merchandising Security & Reporting Structure function.

In all instances, when the user has invoked the agreement maintenance screen or detail forecast screen, the invoked screen will display the information for the selected agreement or detail forecast.  When the user closes the invoked screen, the user is returned to dashboard.

The FP&A dashboard will display statistics on specific agreements, detail forecasts and AP Rebate Programs.  The dashboard will also provide a method for FP&A to select a specific plan or forecast version and drill to a recap of the buyers’ revisions.

The top row of the ‘AP Rebate Programs Column’ will show the number of with a status of ‘Pending AP Approval’ that are within 15 days of the begin date. These 2 special statuses will provide a way for FP&A to identify ones that might need to be worked before others to ensure they are ready when earnings are scheduled to begin. The 30 days and 15 days are configurable options.

In the event that the Accounts Payable group receives a check from a vendor but the agreement status has not yet reached the status of ‘Pending AP Approval’, the user will have the option to search for existing forecasts (using the ‘Search screen in shown in Forecast Search section below) with other statuses to create an AP Rebate Program so the check can be posted.

The following business rules will apply:
There are some rebate types that allow entry of zero dollar rebates (TPR & Markdown Core)on the Merchandise side; however, those will never be presented in the AP Approval Queue (the logic that changes the agreement status will determine if an agreement has a zero dollar amount, and rather than changing the status to ‘Pending AP Approval’ it will be systemically changed to ‘Negotiation Ended’) so that AP does not have to deal with these ‘zero rebate agreements’.
If AP marks an agreement as “Rejected by AP”, AP user will be prompted to enter a comment and a notification will appear on the appropriate buyer’s dashboard.    
A/P will have the opportunity to accept or ignore amounts for all forecast programs with a source of ‘No Source’ including:
No Source - Lump Sum
No Source – Period
There are some rebate types that require a special interface between Merchandising and AP to allow parts of them to come into AP Approval Queue multiple times. They are as follows:
DG Racing: Each period the Lump Sum amounts for DG racing agreements will be presented so that AP can bring them in based on the calculated lump sum amounts from the DG Racing Master-file.
Markdown Core or Markdown Non-Core Agreements:
Will not be shown in the AP Approval Queue until the 5 days (set by a system parameter) after the markdown cadence date.  
Agreements with a contract type of ‘Cadence’: SKU’s will be presented in the appropriate period of their cadence date. An additional AP Rebate Program will be created for each SKU group for a vendor / Planogram and will be systemically linked to the 1 agreement ID and will be linked using the AP VRS linking functionality.
Agreements where the buyer has indicated that # of stores or coolers is variable:  
Will also be presented each month along with the appropriate # of stores based on the criteria selected by merchandising (A flag will be added to the merchandising module to indicate if store / cooler counts should be passed to AP) 
The # of stores or # of coolers will be different each month thus will need a new AP Rebate Program. 
The # of stores will be systemically determined each period based on selection criteria used for the agreement (i.e. Region, Store Layout, State, etc.).
When buyer has selected store count & variable the following occurs:
1 entry will be in the AP Approval Queue for each period during the contract
Subsequent period entries show up in the AP Approval Queue at the beginning of each period
# of stores will be obtained from the store master file at the time period agreement information is brought into AP based on the criteria (as selected by user at the time agreement was created) 
Each period’s AP Rebate Program will be linked to the agreement ID 
The agreement status will be set to ‘Assigned to AP Rebate Program’ from the time the original agreement is brought into AP
A notification will be sent to the A/P Approval Queue, if the buyer is revising an agreement that is attached to an AP Rebate Program but the revision is underway and hasn’t yet reached the A/P Approval Queue. It will be seen as ‘Agreement Revised by Buyer Not Yet to Approval Queue’. The purpose for this is to alert AP that a revision to an existing AP Program is in progress. Once it is revised it will naturally come through the approval queue again.
Supporting SharePoint documents attached to a Rebate Agreement will be eligible for archive when the associated Merchandise Rebate Agreement is purged; however, they will be available in an archive (which can be accessed by Dollar General IT) indefinitely.

 5.4.3 AP Approval Queue – A/P Rebate Program Interface 

The following business rules will apply (further definition might occur during the development process as needed to ensure actual earnings calculations can occur correctly).
:
If the forecast program contains multiple agreements for the same vendor, a separate AP Rebate Program will be created for each duplicate vendor and they will be systemically linked via the AP Linking functionality. 
AP Rebate Description rules: 2 digit year /2 digit period - Vendor Name – AP Rebate Type Code (example: 10/02 P & G VOLX). If there is already a rebate program with that description, a suffix will be added to ensure it is unique. 
In the merchandising system, a buyer will have the ability to note if a forecast program applies to 100% of classes in a department or all SKU’s in a class. If it does, then the rebates will be systemically changed to the department or class level and in the case where there is SKUs information entered SKU’s will become informational only. The purpose for this is to create a forecast at the class level yet not limit the actual earnings calculations to specific classes or SKUs.  
Exclusive SKU Rules:
If source is sales and SKU’s listed are not exclusive to this vendor, note this condition as a warning message to A/P so they can take action if desired. The A/P extract will gather sales information for all primary and sub SKU’s but this situation might necessitate manual intervention to adjust the earnings so they do not invoice a vendor for sales not included in the rebate.
If the source is sales and the SKU is not listed as exclusive to the vendor, a comment will be created for the AP Rebate Program stating the SKU is not exclusive to the vendor.
If the source is either NS-Lump Sum or NS-Period the following rules apply:
AP will have the ability to view and either import lump sum amounts into the AP system or not.
If they are imported, the amounts will have the sign changed as they are brought into the AP Rebate Program because they are a direct translation into earnings.
Lump Sum amounts will be processed in period of From Date regardless of extract dates
The allocation percent by class as determined from the period allocation method used for forecasting will be passed to the AP Rebate Program for allocation purposes.
Amounts will be presented to AP at the class level, not at the SKU level since they cannot come into the AP system at the SKU level and SKU’s will be changed to be informational only.  
Normally they will feed into PMU field except for the following:
For Ad Print Coop which both the COGS (PMU) and expense portions (Coop) can be brought in from the AD Print Coop Master-file which will be updated by the marketing Dept
If the agreement spans fiscal years, then the appropriate part of the lump sum will be brought in as Margin.
If the agreement’s begin date is in prior periods, the portion of the lump sum in the prior periods will be brought into the current period. 
If A/P changes a NS-Lump Sum amount or a portion of an amount from PMU to Margin, the Forecast Program status will be marked as ‘Revised by A/P’ and a notification will be sent to the appropriate buyer’s dashboard.
The ‘From / To’ Dates of an AP Rebate Program::
The AP Program Extract ‘From / To’ Extract Dates will be the same as the Forecast Program ‘From / To’ Dates.
If the Forecast Program ‘From’ Date is today or in the past, the AP Rebate Program ‘From’ Date will be the date the AP Rebate Program is created.
If the Forecast Program ‘From’ Date is in the future, the AP Rebate Program ‘From’ Date will be the Forecast Program ‘From’ Date (same as ‘From’ Extract Date).
The AP Rebate Program ‘To’ Date will always be the same as the ‘To’ Extract Date.

6.1 Buyer - Reports

6.1.1 Deal Variance Report

The Deal Variance Report compares the actual earnings to the ‘original’ forecast at the deal level forecast (detail forecast level) for a selected fiscal period, The report will compare actual to forecast at 4 different levels through the selected period: (1) fiscal period, (2) quarter to date (quarter of period selected), (3) year to date and (4) life to date. 7.0  FP&A Forecast 

FP&A and the merchants compare the rebates’ forecast to plan, LY and actual earnings to this year’s plan, last year’s actual earnings and forecasted earnings during the close of each rebate accounting cycle through the remainder of the year. 

Using the comparisons, FP&A and the merchants adjust the forecasted rebate earnings.

At the close of the rebate accounting period, forecast dollars are replaced by the current period’s actual rebate earnings.

 7.3 FP&A Forecast – Forecast Uploads

FP&A Forecast – Uploading Forecast Data

Forecast data can be created or adjusted using a .csv file upload.  

The upload will adhere to the following rules:
The upload will include an existing forecast name/version.
Data from a .csv upload will be an FP&A adjustment element of the forecast.
The file name will agree with the file name entered in the file name entered in the forecast component.
A department is required in each record and must exist in Dollar General’s merchandise hierarchy.
The rebate type is required and must exist in the Merchandise Rebate Type file of the Vendor Rebate System.
Vendor, rebate category and class are optional; if these fields are on the records, they must exist in Dollar General’s master files. 
If the vendor is not present, the dollars will be added to the “No vendor” vendor.
If the class is not present, the dollars will be added to the “No class” class.
If the rebate category is not present, the dollars will be added to the “No rebate category” rebate category.
The amount must be numeric or null.  If the value is zero, zero will be used in the process.  If the field is blank or null, no value will be used in the process. 
If the upload forecast component option is “revision”, the forecast numbers will be replaced. If the upload forecast component option is “add-to” the forecast numbers will be adjusted.    
Only periods defined as comprised of upload data on the upload components will be impacted.  If data exists in other columns in the .csv file, the data will be ignored. 

If exceptions are found, the upload will not occur and exceptions will be reported.  The exception report will include details on the record and the exception condition to aid in resolution. 

FP&A is responsible for the rebate forecast.  Therefore, access to the Rebate Forecast upload will be controlled through the assignment of the appropriate user group to the .csv upload.

8.0  Rebate Forecast Cycle Close

Business Rules:

During the Rebate Forecast Cycle Close, Dollar General performs 4 major business processes:
Reviewing variances between preliminary rebate earnings and the closing period’s forecast for corrective action
Posting of actual earnings 
Actualizing current period and recasting remaining open period’s forecast in the detail forecasts
Developing and distributing the next period’s FP&A forecast

The close process uses FP&A’s “Current” forecast to compare the closing period’s earnings to forecast; therefore, FP&A will create a new “Current” forecast only after AP has closed the period, the detail forecasts have been actualized and their future forecast periods recast.

The FP&A Rebate Forecast Close module of the Vendor Rebate System will be used during the close cycle to assist in the organization’s close processes. 

8.1 Rebate Forecast Cycle Close

To begin the Rebate Forecast Cycle close, the user will press the “Begin Rebate Close” button.   The system will prepare for reporting the closing month’s rebate performance by creating a working copy of the FP&A forecast marked as “current”. This must occur before getting preliminary actual earnings.  

Pressing the “Get Preliminary Actuals” button will bring the earnings recorded in the Vendor Rebate System Accounts Payable module into the Rebate Forecast for Preliminary Actuals vs. Forecast reporting.  The preliminary earnings (recorded in the AP module as PMU) will be recorded in the Merchandising module for analysis at both FP&A forecast and detail forecast levels (will show on detail forecast screen).  The earnings dollars which AP has recorded as Margin dollars (GL account 429460) will also be recorded for reference and review.  This must occur before producing the preliminary reports.  This can be done multiple times.  

The user will request the system create the preliminary actuals to forecast report and inquiry screen by pressing the “Create Preliminary Reports” button. The actual dollars will be the dollars from the “Get Preliminary Actuals” process above; the forecast will be the closing period’s forecast.  Both the “Begin Cycle Close” and “Get Preliminary Actuals” processes must have been completed.  “Get Preliminary Actuals” can be done multiple times in the closing process until AP has closed the period.

The user will open the “Rebate Actuals and Variance Analysis” form to the merchants by pressing the “Open Analysis to Merchants” button.  Until this has been done, access to the form will be limited by user role to FP&A and AP.  This will be done one time for each cycle close period.
Notes:
The closing period on the form will default to the next closing fiscal period unless the system date is within x days of a fiscal period close date.
The number of days used to determine which closing period to display will be a system configuration.
The fiscal period’s “Open Analysis to Merchant” status will default to “locked” when a fiscal period is open and closing.  The status will be changed to “unlocked” to allow the merchants access using the “Open Analysis to Merchants” process.
The user may choose a closed period to view the closed period’s information.
Options which are no longer valid for a closing period will be grayed out and not operational.
All buttons except “Begin Cycle Close” will be grayed out for an open period
“Create Preliminary Reports” will not be available until “Get Preliminary Actuals” has been run at least one time
All buttons will be grayed out for a closed period
“Begin Cycle Close” will create a “Final” version of the “Current” forecast (leaving the “Current” forecast unchanged) for close processing.  This will be done systemically.

8.4 FP&A Forecast: Finalize Actuals 

When the Accounts Payable module of the Vendor Rebate System closes the period, the Merchandising module will be triggered to:
Replace the closing period’s forecast with actual earnings in the final version of the FP&A forecast.  The system will archive the actualized forecast
Actualize the detail forecasts’ closing period
Recast the detail forecasts’ future periods’ forecast as defined by the agreement’s reallocation parameter or the agreement’s merchandise rebate type reallocation parameter
Mark the closing Rebate Period’s closing status “Closed” 
Capture the AP close date for ease of reference and display on the Rebate Forecast Cycle Close form

8.5 Actualize detail forecasts

When the Accounts Payable module has closed the period, the system will replace the forecast for the period just closed with the period’s actual earnings and mark the closing period as actualized.  

The system will determine whether the detail forecast’s remaining out months (future fiscal periods) will be reallocated using either the merchandise rebate type’s system allocation flag or the detail forecast’s system reallocation flag.  When the detail forecast has a system reallocation flag, the detail forecast flag will override the merchandise rebate type’s reallocation flag.

The system will perform one of six reallocation processes based on the reallocation flag:

1)  Do not re-allocate if earnings are over or under the forecast

2) Reallocate the difference by adjusting the next period’s forecast by the difference (even if after end date) if variance is +/- by x % (set as system parm); if the difference in the forecast is within the tolerance, do not adjust the next period’s forecast.

3) If actuals come in over the forecast do not re-allocate for future periods; if actuals come in equal to or under, perform the reallocation process detailed below

4) If actuals come in under the forecast do not re-allocate for future periods; if actuals come in equal to or over, perform the reallocation process detailed below


5) If the program end date is past, do not reallocate; if end date is not past, re-allocate to remaining future periods using the reallocation process detailed below


6) Always re-allocate, even if the end date is in the past, using the reallocation process detailed below


Reallocation process:

Future period’s detail forecasts will be adjusted.  The remaining estimated earnings dollars will be allocated to the remaining future forecast periods by each remaining period’s percent contribution to the remaining dollars to be earned.  (see table below for detailed example.)
If the actual earnings exceed the total forecast amount, the forecast will be set to equal the actual earnings and if this is not the last period of the agreement, future period’s forecast set to zero.
If this is the last period of the agreement and the actual earnings are differ from the total expected forecast, the rebate’s total expected forecast will be adjusted to the total actual earnings.
When the total forecast dollars are adjusted, the system will determine if any of the three conditions requiring buyer notification and if these exist, the detail forecast will be marked for notification on the Detail Forecast section of the Buyer Dashboard.  The three conditions are:
Detail Forecast closed lower than overall forecast
VRT actual earnings exceed overall forecast
VRT overall change exceeds the system’s change threshold

The forecast will be removed from the Buyer Dashboard when the forecast has been reviewed and remediated.
Note:
The change threshold requiring buyer review will be a system defined parameter.
The tolerance will be a system defined parameter.

11.0 Archiving / Purging

Archiving and Purging of the Merchandise Rebate Systems FP&A Plans and Forecasts, Agreements and Detail Forecasts will be managed through systemic creation of archival data, user managed data purges and defined retention configurations.

11.1 Archiving

Forecast Program Details
No business need has been defined for creating an archive file of Merchandise Forecast Details past the time of purging. However, there is a business requirement to archive the SharePoint documents attached to Rebate Agreements. This requirement is from the A/P department to satisfy audit and SOX requirements. During the Merchandising process, an attribute will be set for the document stating that is eligible for archive and can be moved to offline SharePoint storage by the Dollar General IT department. 

FP&A Plan and Forecast Archiving
FP&A Final Plans will be maintained on the system for the period defined by the system administrator.  

An archive of the FP&A Forecast at the close of the fiscal period will be maintained systemically through the creation and archiving of “Final” versions of the actualized closing forecast.

Forecast Program Details

Aggregated Receipts File:  Fiscal Year / Fiscal Period / IP Vendor # / Dept / Class / SKU / DC or DS Flag (indicating DC or drop ship receipts) / units / dollars

Aggregated Sales File:  Fiscal Year / Fiscal Period / Fiscal Week # / Dept / Class / SKU / units / dollars

When a rebate applies to specific stores, average receipts or sales per store will systemically calculated by dividing the total sales or receipts for the specified timeframe by the total number of stores open at the beginning date specified (from the Front End Master file) then multiplying the average sales or receipts per store to get the base amount needed for forecasting purposes.

When a rebate is based on receipts and is shorter than 1 period, an average weekly receipts amount will be calculated by dividing the FY period amount by the number of weeks in the appropriate period using the retail calendar to get an average weekly receipts number.

 When a rebate is based on sales and is shorter than 1 week, an average daily sales amount will be calculated by dividing the FY week amount by 7 to get an average  daily sales number.

Glossary

Term 
Definition

Act/Fcst
FP&A data point for analysis: represents dollars at the intersection of fiscal year/period, merchandise rebate type, department and vendor.  For closed fiscal periods, dollars are summarized actual earnings recorded by Accounts Payable for the individual detail forecasts.  For current and future periods, the dollars are forecasted earnings.  

Adv Coop
It is a flat dollar rebate offered to vendors to advertise items on a given block or for specific page allocation within a Dollar General circular.  Advertising Rebates when recorded may offset actual advertising costs; otherwise, they are accounted for as rebates increasing purchase mark-up or PMU.
In the Vendor Rebate system it is a flag that indicates that the Earnings needs to be split into two different accounts thereby driving the need to specify an additional Accounting Control Master entry to be associated specifically with a portion of the earnings. 

AP Rebate ID
A unique “numeric” number that defines a Rebate Program in the AP module of the Vendor Rebate System.  A Rebate Program will have at last one Department and Vendor and be of a specific type.  


Calculated Frequency – Accounts Payable Module
Used with the Flat Amount and the Calculated Periods to derive earnings.  Specifies if the calculation is done by the Period or by the Quarter. 

Calculated Periods – Accounts Payable Module
This is a number used in conjunction with the Flat Amount.  It is divided into the Flat Amount in order to derive earnings. 

Cap Amount
Specifies the total amount of earnings that can be generated during the life of a rebate program. Once the “cap” is reached, earnings will not be incremented even if the rebate program is still active. 

Class
The 3rd Tier of the merchandising hierarchy: Region (aka. Division), Dept, Class, SKU.  It is used to Group SKU’s together.  Some buyers have responsibility for a group of classes within a department.

Contract
Within the Vendor Rebate System, this provides a method to group multiple agreements together for reporting purposes.

Department
The 2nd Tier of the merchandising hierarchy: Region (aka Division), Dept, Class, SKU.  It is used to group classes. Most of the time, buyers have responsibility for an entire department.

Division
1st Tier of the merchandising hierarchy. It is used to group departments.

Earned
This is a reference to monies that have been accumulated based on meeting the criteria specified in an agreement with a vendor. Generally, monies are earned when merchandise has been ordered, bought, received or sold; although, there are other scenarios - both merchandise and non-merchandise related.

Flat Amount
The term “Flat Amount” is a reference to an amount that is keyed in versus one that is calculated.  In terms of the Vendor Rebate System, the flat amount is used in “non-sourced” rebates to indicate the amount that is expected to be earned over the life of the rebate program.  It is typically used in a calculation where it is divided by the “Calculated Periods” to derive a period’s earnings. 

Forecasted Earnings
Estimated earnings.  Done at two levels:
Detail forecast at rebate agreement level by the merchants.
FP&A Forecast at fiscal year/period, merchandise rebate type, department and vendor level: combined effort of FP&A and merchants.   

Margin Adjustments
Adjustments that affect the Prior Year.  These are recorded in the Merchandising Module of the Vendor Rebate System and reported as separate (non-forecasted) dollars.

Merchandise Rebate Type 
(Merch Type)
Types of rebate programs used by FP&A and the merchants for management and reporting.  The Merchandise Rebate Type is related to and differs from the Rebate Type used by Accounts Payable.  The relationship between Merch Types and AP Rebate Types in maintained in the AP module of the system.

No Source
This means that data is not extracted in order to calculate earnings.  A “no-source” rebate program uses Flat Amounts divided by the Calculated Periods to derive earnings. 

On Demand Deductions
A reference to taking a deduction in the middle of the period and not waiting for the system to generate a deduction based on earnings. At the point where the deduction is taken, none, some or all of the dollars may be earned; however, an On-Demand deduction will be considered “prepaid”. 

One Time Agreement
Agreements with a begin and end date that are not perpetual or renewable. 

Orders Written
This is a reference to Purchase Orders that have been sent to the Vendor regardless of whether merchandise has been received.  It is one of the “sources” of extracted data used to calculate earnings in the Accounts Payable module. 

Other Coop
The Other Coop amount is to be used when part of a rebate covers expenses which are not for advertising.

Perpetual Agreement
An agreement that is ongoing until cancelled. Agreements of this type will be automatically extended each year.   

PMU
Purchase Mark Up – This is a measure of buyer’s performance. Earnings for the current year that come from the Vendor Rebate System directly affect the PMU.  PMU earnings are recorded as actual earnings in the actualized forecasts. 

POG
Plan-o-Gram – in the context of the Vendor Rebate System, a POG is used to identify associated markdown cadence dates and percentages, the reset date range and classes.  

POS Sales
This is a reference to merchandise sold to the end consumer.  The data is acquired from the POS.  It is one of the “sources” of extracted data used to calculate earnings.

Prepaid Checks
A check that is received prior to any earnings having been generated.  Technically it is not the check that is prepaid but the amount of money it represents. It may be that only a portion of a dollar amount represented by a check is prepaid.  

Prepaid Deductions
A deduction taken prior to earnings having been generated.  In the Vendor Rebate system, Prepaid Deductions are synonymous with On Demand Deductions. 

Prepaid Rebates
A term applied to rebate programs that have had some from of prepayment (deduction or check) made against them.

Rebate Adjustment
A reference to the ability to manipulate the numbers internally generated by the Vendor rebate System. Specially, the ability to adjust Accruals, Reclassifications and Payments.  

Rebate Setup Parameters
The pieces of information that are required to generate an agreement, the detailed forecast and are needed to setup an AP Rebate Program. The general ones are begin/end dates, Merch Type, source for calculating actuals, vendor, department and class.

Rebate Type
In the broader context, the type is a way to classify rebate programs (Ex. Volume DS, Volume DC, Scan Back, etc.).  In the AP Vendor Rebate System, types do classify rebate programs but more specifically, they define the source and the rules used when setting up a rebate program. 

Rebate Type Category
A mechanism used to group Rebate Programs regardless of the type. This helps implement the idea of a “broader” Rebate program such as Back to School.  

Reclass
An abbreviated version of the word reclassification.  It is a reference to moving from a “prepaid” state to an “earned” state.  Monies are “reclassified” when they have been prepaid but now have been earned.

Renewable Agreement
An agreement that is subject for renewal, The buyer will be notified by the VRS when agreements of this type are approaching their end date so they can renegotiate the rebate agreement.  

VRA – Vendor Rebate Agreement
A PDF generated by the system that includes rebate parameters and the legal information. This will be the serve as the contract between Dollar General and the vendor. 

VRT – Vendor Rebate Template
A PDF generated by the system that includes rebate parameters but no legal information. This will be used to provide a consistent document that contains information about an agreement but will not serve as the agreement between Dollar General and the vendor.

Extract Dates and PMU Lump sum can be uploaded with the Rebate upload

(Quantity * Amt per Unit) OR (Expected Quantity * Cost * % of Cost)

Rate, Sides and Store can be uploaded with the Rebate upload
