1. Vision & the timing argument

The new VRS is a single-canvas, role-aware, AI-augmented platform that replaces an Oracle Forms application built ~30 years ago. It preserves Ken Banks' logical data model intact while completely reimagining the surface — entry through a vendor bubble field, role-lens views over a unified record, and an agentic AI advisor (Vera) that synthesizes earnings, calculations, and 1010 transaction history to surface revenue intelligence the legacy system has no way to produce.

Five governing principles

  1. Role determines lens, not navigation. AP analysts, buyers, DMMs, and managers all enter the same canvas; their role calibrates what they see and which actions are available. The MDSE/AP split disappears as a UI concept and becomes a role-filtered view of unified data.
  2. The bubble field is the entry point, not a menu. 600–900 vendor bubbles, where size = annual rebate earnings, color = health, pulsing ring = approval queue state. Period and merch-type filters reconfigure the same canvas for different users and questions.
  3. Vera is a first-class layer, not a feature. Persistent role-aware chat overlay on every screen, with both reactive Q&A and proactive nightly digests. Architected on the proven "Ask Frosti" two-phase pattern from GMCC.
  4. 1010 transaction data is a primary citizen. Nightly materialized summaries of DG's billions of receipt / sales / drop-ship records power pace-to-target, tier-proximity, YoY variance, and uncaptured-opportunity insights — capabilities the current system has no way to surface.
  5. Ken's data model survives. The engine doesn't. Logical schema (program → vendor → dept → calculation), source codes, tier types, four earnings components, calculation state machine — all preserved. Oracle / PL/SQL / Forms — replaced.
The window
Three conditions converge in 2026 that do not converge again. Ken Banks — original VRS author, 30+ years on the system — retires January 2027. Oracle Forms is end-of-life and shrinking the talent pool by the year. TPG Partners has institutional VRS expertise ready to direct at a greenfield build. The knowledge-transfer window closes when Ken does. The 8-week prototype timeline is calibrated against this.

External integrations — heterogeneous, and not as "external" as it looks

Per Ken's clarifications (May 2026, including a late follow-on), VRS doesn't sit at arm's length from DG's other systems — the VRS schema lives inside the RSL Oracle database instance, with Sales, Receipts, Drop Ship, and other operational data in sister schemas on the same DB. Master data (Vendor Master, Dept Master, Class Master) is exposed via Oracle materialized views, also intra-instance. Ken draws a sharp line between true interfaces (state-changing outbound handoffs — file drops or routine invocations) and data access (cross-schema reads, MVs, scheduled bulk feeds from 1010 into VRS Calculate tables). His phrasing: data access "is no different than querying a VRS table."

The post-migration target preserves this heterogeneity. AP moves to Oracle ERP; RSL and GL stay in Lawson; the Vendor Master is bifurcated under the existing materialized view; 1010 stays where it is, walled off from VRS programmatically.

1010 itself — out of scope for VRS direct access

1010 is a separate Oracle database, not part of the RSL Oracle instance, and VRS cannot query it programmatically. Ken's read: heavy ad-hoc use by the AP team; no formal BI tool identified; 1010 effectively serves as DG's analytics warehouse, even if informally. Data flows mostly into 1010 (operational systems including VRS). The selected 1010 → VRS feeds (Discount, NSA, Unsaleables) exist where the data isn't anywhere else and staleness is acceptable. Ownership of the inbound ETL paths sits with another IT group (still unidentified — see Q6 in doc 13).

Implication for the rewrite
Pulling VRS to Postgres is schema extraction from a shared Oracle instance, not standalone-app replatform. The sister schemas VRS reads (Sales, Receipts, Drop Ship, master data) are not moving with VRS. What is today an in-database SQL join becomes a network boundary. The new system needs a federated-read abstraction — oracle_fdw, scheduled snapshot tables, or an explicit ETL — standing in for what is currently a sister-schema query. This is a fundamental shape of the migration, not a network-boundary problem appearing for the first time in production.
Strategic flag — pending Khari conversation
Per Ken: "Everything is being done so that VRS itself does not have to change. It will continue to do things the same way for the foreseeable future." If accurate, DG's modernization plan preserves VRS as-is and the rebuild is TPG-driven rather than DG-driven. The Oracle ERP / Lawson migration timing and the 1010 ownership picture should be clarified with DG-side stakeholders (Khari) before locking in the production target architecture.

3. Data architecture

What the migration actually is — schema extraction, not replatform

Because VRS shares an Oracle instance with RSL and the operational sister schemas it reads (see §2), the rewrite is not "lift one app off Oracle and put it on Postgres." It is extracting the VRS schema out of a shared Oracle box whose neighbours stay where they are. Every cross-schema read VRS does today — Sales, Receipts, Drop Ship, master MVs — becomes a federated read in the new architecture. The 1010-staged feeds (Discount, NSA, Unsaleables) keep their bulk-load shape; the sister-schema joins are the part that needs new mechanics.

  • Federated-read targets: Sales, Receipts, Drop Ship, Vendor Master MV, Common Area master tables.
  • Candidate mechanisms: oracle_fdw for queries that can tolerate live cross-DB latency; scheduled snapshot tables in Postgres for the high-volume / repeatedly-queried slices (the same pattern the legacy already uses for Discount via 1010 staging); explicit ETL into analytics_summaries for things consumed by Vera.
  • Decision needed pre-cutover: which of the sister schemas the VRS Extract reads also stay on Oracle long-term vs. get migrated themselves. Ken's read is they stay; that's the operating assumption until a DG-side stakeholder (Khari) confirms.

What we keep from Ken's model

  • The three-level hierarchy: RebateProgram → RebateVendor → RebateVendorDept → CalculateResult. One calculation per (program, vendor, dept, period).
  • Source-driven AP architecture: 11 source codes (R, S, D, B, F, C, P, N, E, Q, T) plus X (Store Allocations, added 2018). Sourced rebates pull external data via Extract; No-Source rebates use direct entry or formulas. Source is locked at program creation.
  • Four earnings components: PMU (Purchase Mark Up), Margin (prior-year recorded current-year), Adv Coop, Other Coop. The component determines RSL/GL routing.
  • Four tier types: E (Excess), P (Pre-defined Ranges), L (Predefined Levels), G (Growth).
  • Calculation state machine: Calculated → Reviewed → Approved → Batched → Distributed; revertable up through Batched.
  • Period model: YEAR=0 / PERIOD=0 for the open period; real values written at Finalize.
  • Rebate Type as the Source × Merch Type intermediary.
  • Accounting Control Master — RSL/AP/GL routing rules, transaction types, cost centers.

What we change — the Oracle Forms denormalization, dropped

The legacy CALCULATE_RESULT and REBATE_VENDOR_DEPT carry ~190 and ~130 columns respectively, with most of that count being denormalization artifacts. We collapse those in the new schema:

  • SBT / NSBT triplication. Every earnings field exists in three forms: base, SBT_, and NSBT_ — with the invariant SBT_x + NSBT_x = x. Ken's own doc flags this as a maintenance nightmare. We model the SBT classification once (a flag or dimension) rather than tripling every column. The base value is derived; SBT and NSBT views are computed when needed.
  • Time-horizon rollups (cur_, ytd_, ely_, ltd_). These are aggregate views of the same underlying period-grain data. Compute on read, don't store on write.
  • Review-screen rollups (PM_* = PMU+Margin only; AO_* = Adv Coop+Other Coop only). Same — derive, don't store.
  • Vestigial AADJ accrual adjustments. Per Ken: only ONE has ever been created in the entire life of VRS. Drop the columns; if AADJ ever needs to come back, add it then.
  • REBATE_VENDOR_DEPT as a calc cache. The legacy table mirrors the most-recent CalculateResult to power the Vend/Dept tab. We don't need this in Postgres — query the source of truth and cache at the application layer if needed.

Sign convention — handled at ingest

Earnings are natural negative in DG's data (negative = vendor owes DG). The new system normalizes on ingest: store as positive scalars with explicit type (earnings_owed vs earnings_paid), and let the UI render the semantic. The current bubble-field metric for "annual rebate earnings" sums finalEarnings; with real data we'll either negate on read or store the absolute value — decision pending the migration ETL design.

1010 access pattern

The 1010 Oracle database holds the longitudinal vendor-receipt / sales / drop-ship history at warehouse scale (DG bulk-stages ~600M rows of Discount data alone from 1010 into VRS daily). It stays where it is — it is not migrated, and VRS cannot query it programmatically (another IT group owns the inbound ETL paths). The new VRS builds a read path that mirrors the legacy 1010-staging pattern at a different granularity: a scheduled job summarizes 1010-derived data into materialized summary tables in the VRS Postgres database at vendor × dept × source × fiscal_period grain, suitable for Vera and pace-to-target views. The application never queries 1010 directly.

This makes the analytical layer fast (Postgres-native, indexed), independent of 1010's availability or schema, and trivially testable with mock data in the prototype. A separate ingest path preserves the bulk-staged operational feeds (Discount, NSA, Unsaleables-true-up) that the legacy E&C depends on — those are not Vera-derived analytics; they are operational source data without which specific source-types cannot calculate.

4. Information architecture

The bubble field (entry point)

Landing page after login. A D3 force-simulated field of vendor bubbles — one per active vendor, ~600–900 in production scale. Visual encodings:

  • Size → annual rebate earnings. Power-law distribution produces a clean, readable canvas.
  • Color → health. Green = clean. Amber = needs attention (pending approval, calc nearing review deadline, 1010 anomaly flag). Red = overdue calc, unresolved invoice, data-quality flag.
  • Pulsing ring → approval-queue state. A vendor with an agreement at Pending AP Approval gets a pulsing outer ring — the queue becomes a notification, not a destination.

Filters: Merch Type, Source, Category, Analyst, Period, Status, Queue-pending, "My Programs." Period time-slider lets users scrub fiscal history and watch the field evolve.

The vendor record (role-lens view)

Click a bubble → unified vendor record. Six tabs; role determines which is active by default and which actions are available in the context menu.

Workflow surfaces

  • Period close checklist — guided 6-step workflow: E&C status → Reviewed → Approved → Batched → Distributed → Finalized. Live completion percentages, blockers surfaced inline. Replaces the legacy 5-form sequence.
  • Approval queue — surfaced as the pulsing-ring on the vendor bubble. Click → inline review. Approve creates the Rebate Program in the same operation; Reject sends back to MDSE with reason code. No separate destination.
  • Maintenance — merch types, rebate types, categories, vendor master, Accounting Control Master, parameter table, security & roles. Less-frequent admin destination.
  • Reports — legacy reports preserved during transition: VRS Export, History Extract, Earnings Summary by Merch Type, Batch Detail.
  • Ask Vera — persistent overlay drawer, accessible from every surface, role-aware, conversation memory ~5 turns.

Earnings semantics

Earnings are negative-natural (negative = vendor owes DG). The full earnings figure for a period sums:

CURR_PMU + CURR_MAR + CURR_ADV + CURR_OTH
         + CURR_PMU_EADJ + CURR_MAR_EADJ + CURR_ADV_EADJ + CURR_OTH_EADJ

(The EADJ terms are earnings adjustments — manual interventions on the Adj tab. AADJ accrual adjustments exist as columns in the legacy schema but are vestigial — only one has ever been created.)

6. Vera architecture

VERA = Vendor Earnings & Rebate Advisor. A persistent role-aware AI advisor accessible from every screen. Modeled on the proven "Ask Frosti" pattern from TPG's GMCC product, adapted to the VRS domain where the financial stakes are an order of magnitude higher.

What Vera knows

  • The full rebate-program database — agreements, tier structures, calculation history
  • 1010 transaction data via materialized summaries (longitudinal vendor performance)
  • Accounting Control Master — routing rules, transaction types, cost centers
  • Invoice aging
  • Calculation history — which programs calculate cleanly, which have recurring adjustments
  • The relationships between all of these, simultaneously

Capabilities

CapabilityExample
Operational Q&A"What's open on P&G for P05? Which of my programs haven't had an E&C run this period?"
Tier pace & what-if"Show me every vendor approaching a tier threshold this period. What would our earnings look like if we restructured HBA from flat 2% to a 2/3/4% tier?"
Uncaptured-opportunity detection highest-value"Three vendors in HBA participated in ad co-op last year but have no ADVCOOP agreement this year — that's roughly $280K in potential earnings not under agreement."
Anomaly explanation"Energizer receipt volume in Dept 018 dropped 31% YoY in P04 — possible causes: distribution shift, planogram change, data-quality issue."
Negotiation rationale drafting"Prepare a briefing on P&G before our renewal meeting next week."
Renewal calendar"Seven agreements representing $4.2M in annual earnings expire within 90 days — three have no renewal discussion on record."

Two-phase pattern

  1. Intent routing — Vera identifies what data is needed and which stored procedures or 1010 query materializations to invoke. The ~200 VRS stored procedures become the query layer; the 1010 materialized summaries add analytical depth.
  2. Synthesis — Vera executes the selected queries, injects the results as context, and formulates a natural-language answer, recommendation, or what-if analysis.

Role context, assigned programs, and analyst code are injected automatically into every Vera session. A buyer asking about P&G gets a negotiation-oriented answer; an AP analyst asking the same question gets a calculation/invoice-oriented answer. Conversation memory: last ~5 question-answer pairs.

Proactive surfacing

Beyond reactive Q&A, Vera runs a nightly analysis pass per analyst and surfaces flagged items — tier-proximity alerts, anomalies, expiring agreements, uncaptured opportunities — as a digest available on the bubble-field landing page.

Provider abstraction

VeraService exposes a stable interface over a swappable provider. Prototype: AnthropicProvider (Claude). Production: AzureFoundryProvider (GPT-4o). The chat UI, session management, role injection, and tool-call orchestration are identical in both modes.

The demo killer-question
Log in as a Buyer, open P&G, and ask Vera: "How is P&G tracking against their HBA tier this period, and what would our earnings look like if they hit Tier 3?" Then log in as an AP Analyst and ask the same question. Same vendor, same underlying data, calibrated answers. That demonstration sells the platform.

The non-throwaway test

Every component, hook, and schema is built to survive intact into the production codebase with real data substituted for mock. The auth provider swaps, the Vera provider swaps, the 1010 mock summaries are replaced by the nightly ETL output — everything else is unchanged. The prototype is not a proof of concept; it is Sprint 1 of the production platform.
